## 一、系统架构概述

```
┌─────────────────────────────────────────────────────────────┐
│                     Electron 主进程                          │
│                                                             │
│  configService.js ─── readSettings / writeSettings /        │
│                        stampModifiedItems / tombstone 工具    │
│                                                             │
│  SyncService.js ─── push / pull / sync / clearCloud /       │
│                      _mergeConfigs / _runExclusive /         │
│                      cachePassword / onSyncingChanged        │
│                                                             │
│  WebDAVProvider.js ─ upload / download / list / delete /     │
│                      _request / _parsePropfindResponse /     │
│                      _ensureDir                              │
│                                                             │
│  CryptoManager.js ── PBKDF2 + AES-256-GCM 字段级加密        │
│                                                             │
│  ipc/cloud.js ────── 16 个 IPC handler + broadcastSyncStatus│
│  ipc/settings.js ─── save-settings (含 tombstone + stamp)   │
│  ipc/apiProfiles.js ─ switch/create/delete/rename/duplicate │
└───────────────────────────┬─────────────────────────────────┘
                            │ IPC (contextBridge)
┌───────────────────────────┴─────────────────────────────────┐
│                    Vue 3 渲染进程                             │
│                                                             │
│  cloudSync.ts ────── Pinia store / onCloudSyncStatusChanged │
│  GeneralSettings.vue ── formatSyncError / SYNC_ERROR_I18N_MAP│
│  preload.js ──────── electronAPI.cloudSync* 桥接            │
└─────────────────────────────────────────────────────────────┘
```

**数据流**：
1. 渲染进程通过 `cloudSync` store 调用 IPC → 主进程 `cloud.js` 处理器
2. 处理器调用 `SyncService` → `WebDAVProvider`（HTTP）与 `CryptoManager`（加解密）
3. `SyncService._mergeConfigs` 按 per-item `_lastModified` + tombstone 合并
4. 所有写入路径经过 `configService.stampModifiedItems` 打时间戳
5. 主进程 `onSyncingChanged` 广播 → 渲染端 store 订阅同步状态

## 二、当前存留问题

### 中严重级别

#### ~~N-1~~. ✅ 已修复 — `_mergeConfigs` 中 `lastSyncAt` 兜底逻辑在「从未同步过的旧数据」场景下退化

**修复内容**（三处改动）：

1. **`configService.stampModifiedItems`**：当旧条目没有 `_lastModified` 且内容未变时，补写当前时间戳作为迁移标记，确保后续 `_mergeConfigs` 有时间参考。
2. **`SyncService._mergeConfigs` 合并逻辑**：当本地条目无 `_lastModified` 且 `lastSyncAt === 0`（从未同步过）时，采用「本地优先」策略——不轻易被远端覆盖。仅当本地有时间戳或曾同步过（`lastSyncAt > 0`）时才允许远端覆盖。
3. **`SyncService._mergeConfigs` tombstone 应用**：无 `_lastModified` 的旧条目仅在曾同步过时才受 tombstone 约束，从未同步过的旧数据不会被 tombstone 错误删除。

---

#### ~~N-2~~. ✅ 已修复 — `sync()` 的 `pull` → `push` 之间无原子性保证

**修复内容**（四处改动）：

1. **`SyncService.pull`**：`pull` 成功写入 `lastSyncAt` 的同时写入 `pushPending: true` 标记（仅当有远端配置合并时）。若 `push` 前崩溃，重启后可据此自动补推。
2. **`SyncService.push`**：`push` 成功后 `delete updated.cloudSync.pushPending`，清除标记。
3. **`SyncService._checkPushPending`**：新增私有方法，检测 `pushPending` 标记并异步执行 recovery push。返回 `true` 表示触发了补推，`false` 表示无需补推。
4. **`SyncService.startAutoSync`**：grace period 后优先调用 `_checkPushPending()`，若有待补推则跳过常规 `_doAutoSync()`，优先保证远端一致性。

**附带重构**：`configService.js` 将 `require('electron')` + `SETTINGS_FILE` 从模块顶层常量改为延迟计算 `getSettingsFile()`，使纯函数（`stampModifiedItems`、tombstone 工具）可在无 electron 环境下直接测试。

---

#### N-3. per-item 合并粒度下同一 server 的 env 键互覆写

**涉及文件**: `src/main/services/SyncService.js:562-572`

**问题**: `_extractSyncData` 直接复制 `settings.mcpServers` 的浅层对象。合并时整个 server 条目被替换，而非字段级合并。如果两台设备同时修改同一 server 的不同 env 键，后推的会覆盖先推的。

**影响**: per-item 合并策略下，这是设计选择而非 bug。但对用户而言，同时编辑同一 server 的 env 可能导致意外丢失。

**修复方向**: 文档明确说明 per-item 合并粒度；或引入 per-field 的 `_lastModified`（复杂度显著增加）。

### 低严重级别 / 代码质量

#### N-4. `CryptoManager.hashKey` 使用非常量时间比较

**涉及文件**: `src/main/crypto/CryptoManager.js:129-131`, `src/main/ipc/cloud.js:89`

**问题**:
```js
// CryptoManager
hashKey(key) {
  return crypto.createHash('sha256').update(key).digest('hex')
}
// cloud.js verify-password
const valid = hash === cs.passwordHash  // 字符串 === 比较
```

`===` 受 timing attack 影响。实际风险极低——密码验证走 IPC 本地通道，不存在远程攻击面，但不符合密码学最佳实践。

**修复方向**: 使用 `crypto.timingSafeEqual(Buffer.from(hash), Buffer.from(cs.passwordHash))`，一行改动。

---

#### N-5. `WebDAVProvider` 每次请求都重新计算 Basic Auth header

**涉及文件**: `src/main/services/cloud/WebDAVProvider.js:227-229`

**问题**:
```js
const auth = Buffer.from(`${this.username}:${this.password}`).toString('base64')
```

每次 `_request` 调用都重新做 Base64 编码。性能影响可忽略，但可以优化。

**修复方向**: 在 `constructor` 中 `this._authHeader = 'Basic ' + Buffer.from(...)` 缓存。

---

#### N-6. `configService.writeSettings` 备份失败会阻塞主写入

**涉及文件**: `src/main/services/configService.js:42-44`

**问题**:
```js
if (fs.existsSync(SETTINGS_FILE)) {
  const backupPath = SETTINGS_FILE + '.bak'
  fs.copyFileSync(SETTINGS_FILE, backupPath)
}
```

`copyFileSync` 失败（磁盘满、权限问题）时异常冒泡，导致后续 `writeFileSync` 不执行。云同步是高频写入场景。

**修复方向**: `copyFileSync` 包裹 try/catch，备份失败仅 warn 不阻塞。

---

#### N-7. 渲染端 `cloudSync.ts` 的 `syncNow`/`pull`/`push` 在 `finally` 中手动置 `isSyncing = false`

**涉及文件**: `src/stores/cloudSync.ts:226-290`

**问题**: store 中 `syncNow`/`pull`/`push` 的 `finally` 块中 `isSyncing.value = false` 和 `status.value.isSyncing = false`。虽然 `onCloudSyncStatusChanged` 事件到达后会覆盖为正确值，但在事件延迟期间或事件丢失的极端场景下，本地 `isSyncing` 会短暂为 `false` 而主进程仍为 `true`。

**修复方向**: 去掉 `finally` 中的手动置位，完全依赖 IPC 推送；或改为 `finally` 中不置位。

---

#### N-8. `commands.js` 不触发 `onSettingsSaved`

**涉及文件**: `src/main/ipc/commands.js`

**问题**: 命令（TOML 文件）的增删改不经过 `settings.json`，也不触发 `syncService.onSettingsSaved()`。当前不在同步范围内（非 bug），但如果未来 commands 需要同步，需要在当前架构外单独设计。

## 三、安全评估

| 方面 | 评级 | 说明 |
|------|------|------|
| 加密算法 | ✅ 良好 | PBKDF2-SHA256 600000 迭代 + AES-256-GCM，符合 OWASP 2023 推荐 |
| 密码存储 | ✅ 良好 | 密码哈希使用 `hashKey(deriveKey(password, salt))`，不存明文 |
| 密码持久化 | ✅ 良好 | 默认 `persist=false`，启用时用 Electron `safeStorage`（OS 级加密） |
| 哈希比较 | ⚠️ 可改进 | 使用 `===` 而非 `timingSafeEqual`（N-4，实际风险极低） |
| 敏感字段加密 | ✅ 良好 | `apiKey`/`searchApiKey`/`cna`/`env.*` 均字段级加密，非敏感字段明文 |
| 传输安全 | ✅ 良好 | 支持 https；Basic Auth over TLS |
| 并发安全 | ✅ 良好 | `_runExclusive` + `_currentSyncPromise` 单一来源互斥 |
